# ADR-014: ビジュアルリグレッションテストを導入しない

## 日付

2026-02-23

## ステータス

採用

## コンテキスト

番組カード（PodcastCard）のレイアウトが設計（横並び）と異なり縦積みになっていることに、実際の画面を目視確認するまで気づけなかった。コードレビューだけでは CSS のレイアウト崩れを検知できないため、自動検知の仕組みを検討した。

## 検討した選択肢

### 1. Playwright ビジュアルリグレッションテスト

- スクリーンショットのベースライン画像と差分比較で、レイアウト崩れを CI で自動検知できる
- 初期セットアップ（ベースライン管理、Tauri コマンドのモック、CI 環境でのブラウザ実行）のコストが大きい
- 画面数が 3 つの現時点では費用対効果が低い

### 2. Storybook

- コンポーネント単体をカタログ化して表示確認を効率化する
- ただし確認は人間の目視であり、`pnpm tauri dev` で直接確認するのと本質的に変わらない
- カタログ管理の手間が増えるだけで、自動検知にはならない

### 3. 導入しない（目視確認）

- 画面数が少ないため、開発時に `pnpm tauri dev` で目視確認すれば十分
- 追加の仕組みやメンテナンスコストが不要

## 決定

ビジュアルリグレッションテストおよび Storybook を導入しない。レイアウトの確認は開発時の目視で行う。

## 理由

- 自動検知が必要なら Playwright のビジュアルリグレッション一択だが、画面 3 つのアプリに対してセットアップコストが見合わない
- Storybook は目視確認の効率化に過ぎず、`pnpm tauri dev` での直接確認と差がない
- 画面数やコンポーネント数が増えた場合は再検討する
